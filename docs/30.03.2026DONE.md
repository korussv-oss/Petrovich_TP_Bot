## Исправление 1 — Ротация логов (`main.py`)

**Проблема:**  
`logging.FileHandler("data/bot.log")` писал все логи в один файл без какого-либо ограничения размера. На момент обнаружения файл `data/bot.log` содержал **~998 000 строк** и продолжал расти. Это создавало реальный риск:
- заполнения диска на сервере,
- деградации I/O при работе с огромным файлом,
- невозможности быстро найти свежие записи в логах вручную.

**Что исправлено:**  
Заменён `logging.FileHandler` на `logging.handlers.RotatingFileHandler` с параметрами:
- `maxBytes=10 * 1024 * 1024` — максимальный размер одного файла 10 МБ,
- `backupCount=5` — хранится не более 5 архивных файлов (`bot.log.1` … `bot.log.5`).

После заполнения текущий файл переименовывается, создаётся новый. Таким образом максимум на диске занято ~60 МБ вместо бесконечного роста.

**Файл:** `main.py`  
**Было:**
```python
logging.FileHandler("data/bot.log", encoding="utf-8")
```
**Стало:**
```python
from logging.handlers import RotatingFileHandler
RotatingFileHandler(
    "data/bot.log",
    encoding="utf-8",
    maxBytes=10 * 1024 * 1024,
    backupCount=5,
)
```

## Исправление 2 — Удалён двойной `continue` (`core/notifications.py`)

**Проблема:**  
В функции цикла проверки комментариев (`run_registry_comments_loop`) в блоке обработки ошибок Jira стояли два подряд оператора `continue`:

```python
                _save_state(state2)
            continue
            continue          # ← этот оператор никогда не выполнялся
        current_count = len(comments)
```

Второй `continue` являлся **мёртвым кодом** — он физически недостижим, поскольку первый `continue` уже передаёт управление следующей итерации цикла. Такой код — явный признак ошибки при рефакторинге: разработчик, скорее всего, добавил `continue` в неверное место, и выполнение программы шло не по той ветке, по которой ожидалось. Код после мёртвого оператора (`current_count = len(comments)`) мог быть намеренно пропущен, что привело бы к некорректной обработке.

**Что исправлено:**  
Удалён лишний дублирующий `continue`. Поток выполнения теперь однозначен.

**Файл:** `core/notifications.py`  
**Было:**
```python
            continue
            continue
        current_count = len(comments)
```
**Стало:**
```python
            continue
        current_count = len(comments)
```

## Исправление 3 — Singleton Bot для уведомлений (`main.py`)

**Проблема:**  
Функция `deliver_to_channel` при каждой доставке уведомления в Telegram создавала **новый объект `Bot`** (а вместе с ним — новую aiohttp-сессию), отправляла одно сообщение, после чего немедленно закрывала сессию:

```python
bot = Bot(token=token, default=DefaultBotProperties(...))
try:
    await asyncio.wait_for(bot.send_message(...), timeout=...)
finally:
    await bot.session.close()  # создаём и убиваем при каждом уведомлении
```

Это означало, что на каждое уведомление (а их может быть десятки в минуту при активной работе) производились:
- установка нового TCP-соединения к серверам Telegram,
- TLS handshake,
- отправка,
- разрыв соединения.

Помимо накладных расходов на сеть, это исчерпывало файловые дескрипторы и создавало нагрузку на event loop.

**Что исправлено:**  
Добавлен глобальный `_notification_bot = None` и вспомогательная функция `_get_or_create_notification_bot()`, которая создаёт `Bot` один раз и переиспользует его для всех последующих уведомлений. При возникновении ошибки сессии singleton сбрасывается (`_notification_bot = None`), и при следующей доставке создаётся заново — это защищает от зависших соединений.

**Файл:** `main.py`  
**Добавлено:**
```python
_notification_bot = None  # глобальный singleton

async def _get_or_create_notification_bot():
    global _notification_bot
    if _notification_bot is None:
        from aiogram import Bot
        from aiogram.client.default import DefaultBotProperties
        from aiogram.enums import ParseMode
        token = (CONFIG.get("TELEGRAM", {}) or {}).get("TOKEN", "").strip()
        _notification_bot = Bot(token=token, default=DefaultBotProperties(parse_mode=ParseMode.HTML))
    return _notification_bot
```
Удалён блок `finally: await bot.session.close()` — сессия больше не закрывается после каждого сообщения.

## Исправление 4 — TTL-очистка словаря antispam throttle (`middlewares/antispam.py`)

**Проблема:**  
Модуль-уровневый словарь `_throttle: Dict[int, float]` хранил метку последнего обращения для каждого уникального `user_id`. Записи в словарь **никогда не удалялись**. За несколько месяцев работы бота словарь накапливал сотни тысяч записей (по одной на каждого пользователя, который хоть раз воспользовался ботом), потребляя память, которая никогда не освобождалась — классическая медленная утечка памяти.

Дополнительная проблема: `_throttle` был модульным (module-level) словарём, что означало невозможность иметь разные настройки cooldown для разных экземпляров middleware.

**Что исправлено:**
1. `_throttle` перенесён из уровня модуля в экземпляр класса (`self._throttle`) — теперь это instance-переменная, что является правильным подходом для middleware.
2. Добавлен метод `_maybe_cleanup(now)`, который каждые `_CLEANUP_INTERVAL = 300` секунд (5 минут) очищает из словаря записи пользователей, неактивных дольше `cooldown * 20` секунд. Это гарантирует ограниченный рост памяти независимо от числа уникальных пользователей.

**Файл:** `middlewares/antispam.py`  
**Добавлено:**
```python
_CLEANUP_INTERVAL = 300.0  # чистим каждые 5 минут

class AntispamMiddleware(BaseMiddleware):
    def __init__(self, cooldown: float = DEFAULT_COOLDOWN):
        self.cooldown = cooldown
        self._throttle: Dict[int, float] = {}       # перенесено в instance
        self._last_cleanup: float = time.monotonic()

    def _maybe_cleanup(self, now: float) -> None:
        if now - self._last_cleanup < _CLEANUP_INTERVAL:
            return
        cutoff = now - self.cooldown * 20
        self._throttle = {uid: ts for uid, ts in self._throttle.items() if ts > cutoff}
        self._last_cleanup = now
```

## Исправление 6 — Защита `telegram_task` / `max_task` от `NameError` (`main.py`)

**Проблема:**  
Переменные `telegram_task` и `max_task` объявлялись только внутри условных блоков:

```python
if telegram_enabled:
    telegram_task = asyncio.create_task(...)   # только если включён
if max_enabled:
    max_task = asyncio.create_task(...)        # только если включён
```

В блоке `finally` они использовались снова через `if telegram_enabled: telegram_task.cancel()`. Это хрупкий код: если в будущем логика условий изменится или появится исключение между созданием задачи и её регистрацией, возникнет `NameError: name 'telegram_task' is not defined`.

Также блок `finally` отменял `status_task` и `comments_task` дважды (в `.cancel()` и в цикле `for t in (status_task, comments_task)`), но не отменял `telegram_task` и `max_task` в цикле `await` — они завершались «на ходу», без ожидания завершения.

**Что исправлено:**  
Переменные инициализируются как `None` до условных блоков:

```python
telegram_task = None
max_task = None
if telegram_enabled:
    telegram_task = asyncio.create_task(...)
if max_enabled:
    max_task = asyncio.create_task(...)
```

Блок `finally` переписан с использованием `filter(None, [...])` — он корректно обрабатывает все четыре задачи единообразно, независимо от того, какие из них были созданы:

```python
finally:
    for t in filter(None, [status_task, comments_task, telegram_task, max_task]):
        t.cancel()
    for t in filter(None, [status_task, comments_task, telegram_task, max_task]):
        try:
            await t
        except asyncio.CancelledError:
            pass
```

**Файл:** `main.py`

## Исправление 7 — Параллельный batch вместо N+1 в STC-задачах (`core/stc_tasks.py`)

**Проблема:**  
Функция `get_stc_assignee_tasks` итерировала по всем `issue_key` из реестра и выполняла **отдельный HTTP-запрос** к Jira (`get_issue_admin_details`) для каждого тикета последовательно. При 100 тикетах в реестре это означало 100 последовательных сетевых вызовов с задержкой ~200–500 мс каждый — итого **20–50 секунд** на один запрос «Мои задачи» у СА СТЦ.

```python
# БЫЛО (N+1):
for issue_key in get_all_issue_keys():
    info = await get_issue_admin_details(issue_key)  # 1 HTTP на тикет
```

**Что исправлено:**  
Добавлены `asyncio` import и семафор на 5 параллельных запросов. Все детали тикетов теперь запрашиваются одновременно через `asyncio.gather`, что сокращает время с O(n × latency) до O(latency / 5):

```python
semaphore = asyncio.Semaphore(5)

async def _fetch(key: str):
    async with semaphore:
        return key, await get_issue_admin_details(key)

results = await asyncio.gather(*[_fetch(k) for k in all_keys], return_exceptions=True)
```

Семафор на 5 запросов предотвращает перегрузку Jira (без него при 100 тикетах все 100 запросов стартовали бы одновременно). Исключения от отдельных тикетов перехватываются через `return_exceptions=True` и не ломают весь список.

**Файл:** `core/stc_tasks.py`

## Исправление 8 — Параллельная проверка статусов с семафором (`core/notifications.py`)

**Проблема:**  
Функция `check_registry_statuses_and_notify` работала по схеме:

```python
for issue_key in issue_keys:
    status = await get_issue_status(issue_key)  # 1 HTTP на тикет
    ...обработка...
    await asyncio.sleep(0.3)  # искусственная пауза
```

`asyncio.sleep(0.3)` на конце каждой итерации — это rate-limiting, который добавляет **0.3 секунды на каждый тикет**. При 50 тикетах — +15 секунд только на паузы, и это не считая реального времени HTTP-запросов. Цикл проверки статусов мог занимать больше, чем сам интервал между проверками (`COMMENTS_CHECK_INTERVAL`), что приводило к постепенному «отставанию» уведомлений.

**Что исправлено:**  
Все запросы статусов батчируются через `asyncio.gather` с семафором на 5 одновременных запросов. Искусственный `await asyncio.sleep(0.3)` полностью убран — семафор сам является rate-limiter'ом. Обработка результатов (отправка уведомлений) выполняется последовательно, что безопасно и не требует блокировок:

```python
_semaphore = asyncio.Semaphore(5)

async def _fetch_status(key: str):
    async with _semaphore:
        return key, await get_issue_status(key)

raw_results = await asyncio.gather(*[_fetch_status(k) for k in issue_keys], return_exceptions=True)

for result in raw_results:
    if isinstance(result, Exception):
        logger.warning("Ошибка проверки статуса: %s", result)
        continue
    issue_key, status = result
    # ...обработка и отправка уведомления...
```

## Исправление 9 — In-memory кэш для пользовательской БД (`user_storage.py`)

**Проблема:**  
Функция `load_user_db()` при каждом вызове:
1. Читала весь `data/user_data.json` с диска
2. Десериализовывала JSON
3. Если шифрование включено — дешифровала каждое поле (`full_name`, `login`, `email`, `phone`) для каждого пользователя

Функция вызывается на **каждое обращение к профилю** — проверка регистрации, создание тикета, уведомление, антиспам. Это O(n) I/O на каждый входящий запрос от пользователя.

**Что исправлено:**  
Добавлен модульный in-memory кэш с TTL 30 секунд и `threading.Lock()` для потокобезопасности:

```python
_cache_lock = threading.Lock()
_users_cache: Optional[Dict] = None
_cache_loaded_at: float = 0.0
_CACHE_TTL = 30.0

def load_user_db() -> Dict:
    global _users_cache, _cache_loaded_at
    now = _time_module.monotonic()
    with _cache_lock:
        if _users_cache is not None and (now - _cache_loaded_at) < _CACHE_TTL:
            return _users_cache   # ← возврат из памяти без I/O
        # ... читаем файл и обновляем кэш ...
```

`save_user_db` немедленно обновляет кэш после записи — не нужно ждать TTL, чтобы изменения стали видны остальным вызовам в том же процессе. Это гарантирует **согласованность** данных при частых save/load в одном цикле.

**Файл:** `user_storage.py`

Каждая из функций реестра (`get_bindings_by_user`, `get_bindings_by_issue`, `get_user_ids_by_issue`, `get_all_issue_keys`, `get_all_bindings`) вызывала `_load()`, которая **каждый раз читала** `issue_binding_registry.json` с диска. _registry_lock = threading.Lock()
```python
_registry_cache: Optional[List] = None

_REGISTRY_CACHE_TTL = 15.0

def _load() -> List:

        if _registry_cache is not None and (now - _registry_cache_loaded_at) < _REGISTRY_CACHE_TTL:
            return list(_registry_cache)  # ← копия из памяти
        ...чтение файла...
```

`_save()` немедленно обновляет `_registry_cache` после записи — добавленная привязка видна всем последующим читателям без ожидания TTL.

## Исправление 11 — Атомарная запись JSON-файлов (`user_storage.py`, `issue_binding_registry.py`)

**Проблема:**  
Обе функции записи (`_save_json` в `user_storage.py` и `_save` в `issue_binding_registry.py`) открывали файл напрямую и писали данные:

```python
with open(path, "w", encoding="utf-8") as f:
    json.dump(data, f, ensure_ascii=False, indent=2)
```

Если в момент записи процесс был убит (kill, OOM-killer, сбой питания, Docker `stop`), файл оставался **частично записанным** — с обрезанным JSON, который невозможно прочитать. При следующем старте бот не мог загрузить ни базу пользователей, ни реестр привязок, и работал с пустыми данными.

**Что исправлено:**  
Реализована атомарная запись через временный файл + `os.replace`. `os.replace` на POSIX-системах — атомарная операция (системный вызов `rename`). На Windows она также атомарна начиная с Windows Vista. Старый файл заменяется **только после успешной записи нового**:

```python
tmp_path = path + ".tmp"
with open(tmp_path, "w", encoding="utf-8") as f:
    json.dump(data, f, ensure_ascii=False, indent=2)
os.replace(tmp_path, path)  # атомарная замена
```

**Файлы:** `user_storage.py` (функция `_save_json`), `core/support/issue_binding_registry.py` (функция `_save`)  
Также добавлен `import os` в `issue_binding_registry.py`.

## Исправление 12 — Retry-декоратор для Jira HTTP (`core/jira_retry.py`, `core/jira_aa.py`)

**Проблема:**  
Все функции `get_issue_info`, `get_issue_admin_details`, `get_issue_status` (через `get_issue_info`) при любой сетевой ошибке или таймауте Jira немедленно возвращали `None`, и пользователь получал ошибку. Кратковременные сбои Jira (~2–5 секунд) приводили к видимым сбоям для пользователей и ложным срабатываниям в цикле уведомлений.

**Что исправлено:**  
Создан новый модуль `core/jira_retry.py` с декоратором `retry_jira`:

```python
def retry_jira(max_attempts: int = 3, base_delay: float = 1.0):
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                result = await func(*args, **kwargs)
                if result is not None:
                    return result
                if attempt < max_attempts:
                    delay = base_delay * (2 ** (attempt - 1))
                    logger.debug("%s: попытка %d/%d, повтор через %.1fs", ...)
                    await asyncio.sleep(delay)
            return None
        return wrapper
    return decorator
```

Функции **создания тикетов** (`create_password_change_issue`, `add_comment`) намеренно не декорированы — повтор создания мог бы привести к дублированию тикетов или комментариев.

**Новый файл:** `core/jira_retry.py`

## Исправление 13 — Персистентное FSM-хранилище (`core/fsm_storage.py`, `main.py`)

**Проблема:**  
`MemoryStorage` хранит FSM-состояния (текущий шаг формы, введённые данные) **только в оперативной памяти**. При любом перезапуске бота (деплой, краш, docker restart) все незавершённые сессии сбрасывались. Пользователь, заполнявший форму создания тикета или проходивший регистрацию, после перезапуска получал пустой экран и был вынужден начинать заново.

**Что исправлено:**  
Создан `core/fsm_storage.py` с классом `JsonFsmStorage`, реализующим интерфейс `BaseStorage` из Aiogram 3 без внешних зависимостей:

```python
class JsonFsmStorage(BaseStorage):
    def __init__(self, path: str | Path = "data/fsm_state.json"):
        self._path = Path(path)
        self._lock = asyncio.Lock()
        self._data = self._load_from_disk()  # загружаем при старте
```

Особенности реализации:
- **Загрузка при старте**: при инициализации данные читаются из `data/fsm_state.json`; если файл повреждён — стартуем с пустым хранилищем и логируем предупреждение
- **Атомарная запись**: `_flush()` использует тот же паттерн `tmp + os.replace`, что и исправление 11
- **asyncio.Lock**: каждая запись (`set_state`, `set_data`) защищена — корректно работает при конкурентных FSM-переходах
- **Без Redis, без доп. пакетов**: работает из коробки в любом окружении

В `main.py` `MemoryStorage()` заменён на `JsonFsmStorage("data/fsm_state.json")`.

**Новый файл:** `core/fsm_storage.py`  
**Изменён:** `main.py` (заменён импорт и инициализация storage)

## Фаза 11 — Безопасность шифрования: singleton Fernet + запрет тихого падения

**Дата:** 30.03.2026  
**Файл:** `user_storage.py`

### 11.1 Проблема 1: Fernet создавался на каждый вызов

**Было:**
```python
def _get_fernet():
    key = os.getenv("USER_DATA_ENCRYPTION_KEY")
    return Fernet(key.encode())  # новый объект на каждый вызов
```
**Стало:** `@functools.lru_cache(maxsize=1)` — объект создаётся ровно один раз.

**Зачем:** Fernet при создании парсит и валидирует ключ (base64-декодирование). При 1000 запросов в минуту это 1000 лишних операций. Кроме того, `lru_cache` гарантирует детерминированность: один ключ → один экземпляр за весь lifetime процесса.

### 11.2 Проблема 2: Ошибка инициализации Fernet — тихий fallback в None

**Было:** `except Exception: return None` без какого-либо лога.

**Стало:** при пустом ключе — `logger.warning(...)`, при битом ключе — `logger.error(...)`, и только потом `return None`.

**Зачем:** если ключ случайно обнулился в env (деплой, опечатка), оператор должен немедленно увидеть это в логах, а не через несколько часов при обнаружении незашифрованных данных.

### 11.3 Проблема 3: _encrypt_value молча возвращала plaintext при ошибке

**Было:**
```python
except Exception:
    return value  # данные пишутся в открытом виде без предупреждения
```
**Стало:**
```python
except Exception as e:
    logger.error("Ошибка шифрования поля — данные НЕ сохранены: %s", e)
    raise RuntimeError("Encryption failed") from e
```
**Зачем:** тихое сохранение plaintext при включённом шифровании — нарушение безопасности. Операция сохранения должна падать явно, чтобы вызывающий код мог обработать ошибку.

### 11.4 Проблема 4: _decrypt_value молча возвращала ciphertext при ошибке

**Было:** `except Exception: return value` — пользователь получал нечитаемую строку вместо данных.

**Стало:** `logger.warning(...)` + `return value` — ошибка видна в логах, сервис не падает (данные уже в БД, поднимать исключение здесь нецелесообразно).

### 11.5 Результат

```
Cache info: CacheInfo(hits=1, misses=1, maxsize=1, currsize=1)
```
Второй вызов `_get_fernet()` берётся из кэша. Тест `_encrypt_value` с битым Fernet поднимает `RuntimeError`.
