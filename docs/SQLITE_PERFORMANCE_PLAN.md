## План максимального ускорения SQLite (WAL/SHM, connect/close, блокировки)

**Контекст:**
- SQLite работает в `WAL`-режиме, поэтому рядом с `data/storage.sqlite3` появляются `storage.sqlite3-wal` и `storage.sqlite3-shm`.
- Эти файлы не “проблема сами по себе”. Проблема — **частые подключения/закрытия**, частые мелкие записи и (на Windows) влияние антивируса на часто изменяемые файлы.

Цель: **минимизировать I/O и блокировки event loop**, сделать latency ровным, убрать “мигание” WAL/SHM и снизить нагрузку на файловую систему.

---

### 0) Как измеряем эффект (обязательно)

Перед изменениями и после:
- **Метрика 1**: время построения “Мои заявки” (кнопка → показ списка).
- **Метрика 2**: p95 задержки UI в MAX (callback → первый ответ).
- **Метрика 3**: количество SQLite операций в минуту (примерно по логам/счётчикам).

Минимально: включить логирование медленных SQLite операций (если > 200мс).

---

### 1) Самый большой эффект: убрать connect/close на каждую операцию

**Проблема:**
Сейчас в `core/storage/sqlite_backend.py` практически каждая функция делает `_connect()` → запрос → `conn.close()`.
На Windows это даёт ощутимую цену, а `-wal/-shm` постоянно создаются/закрываются.

**Решение (рекомендуемый путь):**
Сделать **персистентное соединение**:
- либо 1 connection на процесс + `_DB_LOCK`
- либо `threading.local()` connection (лучше, если активно используем `asyncio.to_thread(...)`)

**Почему это максимально эффективно:**
- перестаём платить цену `sqlite3.connect()` и PRAGMA на каждый запрос
- WAL/SHM перестают “мигать”, уменьшается I/O
- меньше шанс “database is locked”

**Риски/особенности:**
- нужно корректно закрывать соединения при завершении процесса (не критично на практике, но желательно)
- при thread-local важно не использовать connection из другого потока

**Критерий готовности:**
- в коде не осталось массового connect/close для hot-path операций
- WAL/SHM стабильны и не создаются/удаляются десятки раз в минуту

### 2) Настроить WAL, чтобы не было пиков от checkpoint

**Проблема:**
WAL растёт, затем SQLite делает checkpoint, что может давать краткие пики I/O.

**Решение:**
Добавить/проверить PRAGMA в момент инициализации connection:
- `PRAGMA wal_autocheckpoint = 1000;` (подбирать; меньше = чаще, но мельче)
- `PRAGMA busy_timeout = 5000;` (меньше “locked” и повторов)
- `PRAGMA temp_store = MEMORY;`
- `PRAGMA cache_size = -65536;` (пример: 64MB кэша; подбирать)

**Критерий готовности:**
- нет частых lock/timeout на запись
- WAL не раздувается “вечно”

### 3) Максимально снизить число записей: батчить и транзакции

**Проблема:**
Много мелких апдейтов = много fsync/IO.

**Решение:**
- Всё что можно → `executemany(...)` внутри одной транзакции.
- Для state-таблиц (“notification_state”, “pending_password_requests”) — обновлять пачками.

**Статус сейчас:**
- для `issue_notification_state` уже сделан bulk-upsert.

---

### 4) Убрать блокировки event loop от SQLite

**Проблема:**
`sqlite3` синхронный. Любой IO может остановить event loop.

**Решение:**
- все SQLite операции, которые вызываются из async кода, делать через:
  - `await asyncio.to_thread(...)`, либо
  - `aiosqlite` (см. п. 6)

**Статус сейчас:**
- часть hot-path уже вынесена в `to_thread` (см. `docs/PERFORMANCE_PLAN.md`).

---

### 5) Windows: исключить `data/` из антивируса (часто даёт огромный эффект)

**Проблема:**
Defender/корпоративный AV сканирует меняющиеся `*.sqlite3-wal` и может добавлять непредсказуемые задержки.

**Решение:**
- добавить `.../the_bot_rubik/data/` в исключения сканирования

**Критерий готовности:**
- latency UI становится “ровнее”, меньше случайных 1–5 секунд пауз

---

### 6) Опционально для “максималки”: перейти на aiosqlite/DB worker

Если нужно ещё ровнее и чище:

**Вариант A — aiosqlite**
- одно async соединение в event loop
- меньше `to_thread`, проще контроль таймаутов

**Вариант B — DB worker thread**
- очередь операций → один поток с одним sqlite connection
- очень предсказуемое поведение для WAL

**Критерий готовности:**
- ни одна SQLite операция не блокирует event loop
