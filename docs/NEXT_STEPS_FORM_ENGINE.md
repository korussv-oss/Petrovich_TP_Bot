# Напоминание по следующему шагу

Если хотите, следующим шагом можно убрать fallback полностью (после 1-2 дней стабильных тестов) и окончательно зачистить legacy-код путей создания для этих форм.

## Зачем это нужно

- Упрощает кодовую базу: остаётся один источник логики создания заявок (forms_catalog + form_engine), без дублирования путей.
- Снижает риск расхождений: не будет ситуации, когда новая форма работает в новом пути, а часть полей внезапно берётся из legacy-кода.
- Ускоряет развитие: добавление и изменение форм делается через каталог, без правок во множестве файлов.
- Облегчает поддержку и отладку: меньше ветвлений, проще анализировать ошибки Jira и поведение сценариев в TG/MAX.
- Уменьшает технический долг: после стабилизации можно безопасно удалять старые функции и не тащить их в будущие релизы.

## План следующей архитектурной уборки

### 1. Структурировать `core/` по доменам

- Создать подпапки:
  - `core/forms/` — всё, что связано с формами:
    - `forms_catalog.py`
    - `jira_form_engine.py`
    - формовые помощники (`pc_problem.py`, `orgtech.py`, `peripheral_equipment.py`, `network_problem.py`, `electronic_queue.py`, `email_owa.py` и т.п.).
  - `core/integrations/` — низкоуровневые обёртки над Jira/AD:
    - `jira_aa.py`, `jira_wms.py`, `jira_lupa.py`
    - `ad_ldap.py`.
  - `core/domains/` — доменная логика поверх интеграций (WMS, Lupa, сеть и т.д.) — можно вводить постепенно.
- При переносе файлов ограничиться обновлением импортов, без изменения поведения.

### 2. Унифицировать работу с UI-списками форм

- Ввести общий helper, например `core/forms/selectors.py`:
  - `get_options(form_id: str, select_id: str, fallback: list[tuple[str, str]] | None = None) -> list[tuple[str,str]]`.
- Постепенно перевести `keyboards.py` и MAX-flows на использование этого helper:
  - вместо отдельных модулей, которые только читают YAML, вызывать `get_options(...)` напрямую;
  - оставлять отдельные файлы только там, где есть дополнительная логика, а не просто чтение каталога.

### 3. Вынести бизнес-логику форм из TG/MAX-flow в сервисы

- Для сложных форм (ПК, оргтехника, периферия, сеть, электронная очередь, email OWA) завести сервисы в `core/domains/`:
  - пример: `core/domains/pc.py` с функцией `build_pc_form_data(profile, user_input)`;
  - сервис отвечает за:
    - проверку обязательных полей и комбинаций (ветвления по типам);
    - подготовку `form_data` для `form_engine`.
- В `handlers/create_ticket.py` и `adapters/max/*_flow.py` оставить только:
  - сбор FSM-состояния / шагов;
  - вызов сервисной функции;
  - передачу результата в `support_api.create_ticket`.

### 4. Аккуратно убрать legacy Jira-пути

- После того как формы стабильно работают через `forms_catalog + jira_form_engine`:
  - по шагам отключить fallback на `jira_wms.create_*`, `jira_lupa.create_*`, `jira_pc`, `jira_email`;
  - удалить неиспользуемые функции/модули, оставив только общий Jira-клиент и form engine.
- Обновить документацию:
  - пометить `jira_wms.py`, `jira_lupa.py`, `jira_pc.py`, `jira_email.py` как legacy и указать дату удаления.

### 5. Документация и тесты

- Добавить `docs/SMOKE_TESTS.md`:
  - краткий чек-лист по формам (ПК, оргтехника, периферия, сеть, услуги, email, WMS, Lupa, смена пароля);
  - отдельный блок для роли `СА СТЦ`: уведомление, «Мои задачи», статусы, комментарии.
- После каждого архитектурного шага прогонять smoke-чеклист в TG и MAX.
