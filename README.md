# Бот Rubik — актуальная документация

Единый бот поддержки для **Telegram** и **MAX**: регистрация, создание заявок в Jira, «Мои заявки», комментарии, уведомления, админ-функции и роль **СА СТЦ** (задачи по assignee).

---

## 1. Что умеет бот сейчас

### 1.1 Каналы

- Telegram (aiogram)
- MAX (long polling + единый core)

### 1.2 Основные пользовательские функции

- Регистрация / привязка аккаунта по номеру телефона
- Создание заявок через меню `Создать заявку в ТП`
- Просмотр своих заявок (`Мои заявки`)
- Добавление комментариев к заявкам
- Уведомления о смене статуса и новых комментариях

### 1.3 Доступные формы заявок

Формы описаны в `config/forms_catalog.yaml` и создаются через `core/jira_form_engine.py`.

- `Поиск/Сайт` (`lupa_search`)
- `Проблема в работе WMS` (`wms_issue`)
- `Смена пароля` (`rubik_password_change`)
- `Проблема в работе ПК` (`pc_problem`)
- `Оргтехника` (`orgtech_problem`)
- `Периферийное оборудование` (`peripheral_equipment`)
- `Проблемы в работе сети` (`network_problem`)
- `Электронная очередь` (`electronic_queue`)
- `Электронная почта (Owa/Outlook)` (`email_owa_outlook`)

> Для части заявок используется загрузка вложений (до 10 файлов, до 10 МБ каждый).

## 2. Архитектура

### 2.1 Слои

- `core/` — бизнес-логика, Jira/AD, уведомления, form engine
- `handlers/` — Telegram-обработчики (FSM, кнопки, ответы)
- `adapters/max/` — MAX-роутинг и пошаговые flow
- `user_storage.py` — профили пользователей и индексы

### 2.2 Точка входа

- `main.py` запускает Telegram-бота и (если есть токен) MAX-бота
- подключает доставку уведомлений в оба канала
- запускает фоновые циклы:
  - `run_registry_status_loop`
  - `run_registry_comments_loop`

### 2.3 Единый API

- `core/support/api.py` — единый фасад для каналов:
  - `get_main_menu`, `create_ticket`, `get_my_tickets_filtered`, `user_owns_issue`
- все канальные адаптеры идут через этот слой

---

## 3. Роли и права

- **Пользователь** — регистрация, заявки, мои заявки, комментарии
- **Админ** (`ADMIN_IDS` / `ADMIN_MAX_IDS`) — админ-панель, удаление пользователей
- **Админ Лупы** (`ADMIN_LUPA_IDS` / `ADMIN_LUPA_MAX_IDS`) — доступ к Excel-отчёту Лупы
- **Системный администратор СТЦ** (`STC_SA_IDS` / `STC_SA_MAX_IDS`)
  - раздел `СА СТЦ` в главном меню
  - `Мои задачи` (заявки, где пользователь текущий assignee в Jira)
  - карточка задачи: открыть в Jira, установить статус, добавить комментарий
  - уведомления о новых задачах по assignee

## 4. Регистрация и безопасность

### 4.1 Данные и секреты

- Секреты только в `.env` (`JIRA_TOKEN`, токены TG/MAX, AD bind)
- `.env` не коммитится
- рабочие данные в `data/` (в `.gitignore`)

### 4.2 Проверки и валидация

- Централизованная валидация в `validators.py`
- Санитизация Jira-текста: `sanitize_jira_text`
- Валидация `issue_key`: `validate_issue_key`
- Проверка уникальности логина/email/табельного/телефона

### 4.3 AD/LDAP

- Используется для регистрации и проверки истечения пароля
- Смена пароля через бота доступна только при `password expired`

### 4.4 Регистрация в TG/MAX

- После ввода почты бот удаляет сообщение с почтой (требование ИБ)
- Контакт пользователя проверяется на принадлежность текущему пользователю

## 5. Уведомления

`core/notifications.py`:

- уведомления о смене статусов по реестру заявок
- уведомления о новых комментариях (внутренние JSM-комментарии не отправляются)
- уведомления для роли СА СТЦ по назначению assignee

Поддержаны оба канала через `core/support/delivery.py`.

## 7. Конфигурация

Минимум для запуска:

- `TELEGRAM_TOKEN`
- `JIRA_LOGIN_URL`
- `JIRA_TOKEN`

Для MAX:

- `MAX_BOT_TOKEN` (или `MAX_TOKEN`)

Для ролей:

- `ADMIN_IDS`, `ADMIN_MAX_IDS`
- `ADMIN_LUPA_IDS`, `ADMIN_LUPA_MAX_IDS`
- `STC_SA_IDS`, `STC_SA_MAX_IDS`

AD (если используется регистрация/проверки через LDAP):

- `AD_LDAP_URL`, `AD_LDAP_BIND_USER`, `AD_LDAP_BIND_PASSWORD`, `AD_LDAP_BASE_DN`

См. полный пример: `.env.example`.
